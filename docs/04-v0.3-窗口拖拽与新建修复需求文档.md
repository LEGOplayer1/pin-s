# 纸间 v0.3 · 小版本需求文档

> 版本：v0.3 "窗口拖拽与新建修复"
> 负责人：纸间团队
> 状态：需求编写中

---

## 1. 需求背景

v0.2 发布后，用户反馈两个严重问题：

1. **便利贴无法拖动**：用户无法通过拖拽标题栏移动便利贴窗口到桌面其他位置，导致所有便利贴堆叠在默认位置，无法自由排布。
2. **无法创建新便利贴**：点击标题栏"+"按钮或使用 `Ctrl+N` 快捷键后，新便利贴窗口未出现，用户只能使用首次启动的欢迎便利贴。

这两个问题直接破坏了产品核心体验——"每张便利贴是独立窗口、可自由排布"，属于 P0 级缺陷。

---

## 2. 目标与范围

本次小版本聚焦于**修复窗口拖拽与新建便利贴两个核心功能**，不新增任何功能特性。

### 范围内

- 修复便利贴窗口拖拽功能
- 修复新建便利贴功能
- 确保修复后拖拽与新建行为与设计文档一致

### 范围外

- 新增功能（搜索、提醒、自动布局等）
- 条目模式相关改动
- 跨平台适配
- UI 视觉调整

## 3. 问题分析

### 3.1 窗口拖拽失效

**现状代码**：标题栏 `.note-header` 已添加 `data-tauri-drag-region` 属性，底部 `.note-footer` 同样添加了该属性。

**可能原因**：

| 编号 | 原因 | 说明 |
|------|------|------|
| D-01 | 子元素吞噬拖拽事件 | `.note-header` 内的 `.color-dots` 和 `.window-actions` 占据了标题栏大部分面积，这些子元素会拦截鼠标事件，导致 `data-tauri-drag-region` 仅在标题栏极窄的空白区域生效 |
| D-02 | 透明窗口与拖拽区域冲突 | Tauri v2 中 `transparent(true)` 窗口的拖拽区域可能存在已知兼容问题，部分 WebView2 版本下透明窗口的拖拽区域无法正常响应 |
| D-03 | `user-select: none` 干扰 | `body` 上设置了 `user-select: none`，可能在某些情况下干扰 Tauri 的拖拽事件识别 |
| D-04 | 缺少 Tauri 拖拽权限配置 | Tauri v2 可能需要在配置中显式启用拖拽相关能力 |

**预期行为**：用户在标题栏任意非按钮区域按下鼠标并拖动，窗口应跟随鼠标移动。

### 3.2 新建便利贴失效

**现状代码**：前端通过 `invoke('cmd_create_note', { color: note.dataset.color })` 调用后端命令，后端 `create_note()` 函数创建新窗口并写入状态。

**可能原因**：

| 编号 | 原因 | 说明 |
|------|------|------|
| N-01 | 新窗口创建后未获得焦点 | `build_note_window()` 虽然设置了 `.focused(true)`，但新窗口可能在已有窗口下方创建，用户看不到 |
| N-02 | 新窗口位置与已有窗口完全重叠 | `create_note()` 中默认位置计算 `120 + count * 28` 偏移量过小，新窗口与已有窗口几乎完全重叠，用户误以为没有创建 |
| N-03 | 单实例插件回调中创建窗口失败 | `tauri_plugin_single_instance::init` 回调中调用 `create_note()` 时，AppHandle 状态可能未完全就绪 |
| N-04 | IPC 命令参数命名不匹配 | Tauri v2 的 `#[tauri::command]` 使用 snake_case 参数名，但前端 invoke 传入的是 camelCase（如 `color` → `color`），可能存在序列化不匹配 |
| N-05 | 新窗口创建成功但被系统置于底层 | Windows 窗口管理器可能将新窗口置于其他应用窗口之后 |

**预期行为**：点击"+"按钮或 `Ctrl+N` 后，在桌面可见位置出现一张新的空白便利贴窗口，并自动获得焦点。

## 4. 核心用户故事

**US-01 — 拖拽移动便利贴**
> 作为用户，我希望在便利贴标题栏的空白区域按下鼠标并拖动，
> 窗口能跟随鼠标移动到桌面任意位置，
> 以便我能自由排布多张便利贴。

**US-02 — 通过按钮新建便利贴**
> 作为用户，我希望点击标题栏的"+"按钮，
> 立即在桌面可见位置出现一张新的空白便利贴，
> 以便我能快速记录新的内容。

**US-03 — 通过快捷键新建便利贴**
> 作为用户，我希望按下 `Ctrl+N` 快捷键，
> 立即创建一张新的便利贴，
> 以便我在编辑内容时无需移动鼠标即可新建。

**US-04 — 新便利贴不与已有便利贴重叠**
> 作为用户，我希望新建的便利贴出现在一个不会与已有便利贴完全重叠的位置，
> 以便我能立即看到并操作新便利贴。

## 5. 交互规格

### 5.1 拖拽交互

**触发区域**：标题栏（`.note-header`）中非交互元素的区域。

| 区域 | 是否可拖拽 | 说明 |
|------|-----------|------|
| 标题栏空白区域 | ✅ | 主要拖拽区域 |
| 底部状态栏空白区域 | ✅ | 辅助拖拽区域 |
| 色彩圆点 | ❌ | 点击切换颜色 |
| 模式切换按钮 | ❌ | 点击切换模式 |
| 置顶按钮 | ❌ | 点击切换置顶 |
| 新建按钮 | ❌ | 点击新建便利贴 |
| 关闭按钮 | ❌ | 点击关闭便利贴 |

**拖拽行为**：
1. 鼠标在可拖拽区域按下 → 光标变为移动样式（`cursor: move`）
2. 按住拖动 → 窗口跟随鼠标平滑移动，无延迟、无抖动
3. 松开鼠标 → 窗口停留在当前位置，位置信息自动保存

### 5.2 新建便利贴交互

**触发方式**：
1. 点击标题栏"+"按钮
2. 按下 `Ctrl+N` 快捷键（仅在非编辑状态下生效）
3. 右键托盘菜单 → "新建便利贴"
4. 再次启动应用时（单实例回调）

**新建行为**：
1. 新便利贴在当前活跃便利贴的右下方偏移 (40, 40) 位置创建
2. 若偏移后位置超出屏幕可见范围，则回退到屏幕中心位置
3. 新便利贴自动获得焦点并置于最前
4. 新便利贴默认使用与当前便利贴相同的颜色
5. 新便利贴创建后立即保存到 `notes.json`

## 6. 技术实现方案

### 6.1 修复窗口拖拽

#### 方案：前端 JS 拖拽替代 `data-tauri-drag-region`

由于 `data-tauri-drag-region` 在当前环境下存在子元素事件吞噬和透明窗口兼容性问题，改用前端 JS 实现拖拽，通过 IPC 调用后端 `cmd_set_window_rect` 移动窗口。

**实现步骤**：

1. **移除** `.note-header` 和 `.note-footer` 上的 `data-tauri-drag-region` 属性
2. **添加** 前端拖拽逻辑：
   - `mousedown` 在标题栏空白区域 → 记录起始鼠标位置和窗口位置
   - `mousemove` → 计算偏移量，调用 `invoke('cmd_set_window_rect', { x, y })` 更新窗口位置
   - `mouseup` → 结束拖拽，触发保存
3. **优化**：使用节流（throttle）避免高频 IPC 调用导致卡顿，建议 16ms 间隔（约 60fps）
4. **光标反馈**：标题栏空白区域设置 `cursor: move`

**关键代码位置**：
- 前端：`src/index.html` — `<script>` 区域新增拖拽逻辑
- 后端：`src-tauri/src/main.rs` — `cmd_set_window_rect` 已实现，无需修改

**替代方案**（若 JS 拖拽性能不理想）：
- 保留 `data-tauri-drag-region`，但为标题栏内的交互子元素（按钮、色点）添加 `data-tauri-drag-region="false"` 属性，阻止它们吞噬拖拽事件
- 同时增大标题栏高度，提供更多可拖拽空白区域

### 6.2 修复新建便利贴

#### 修复点 1：改善新窗口位置计算

**当前逻辑**（`window_manager.rs` 第 73-74 行）：
```rust
let count = state.notes.lock().unwrap().notes.len() as i32;
let default_x = 120 + count * 28;
let default_y = 120 + count * 28;
```

偏移量仅 28px，新窗口几乎完全被旧窗口遮挡。

**优化方案**：
- 获取当前焦点窗口的位置作为基准
- 新窗口在基准位置右下偏移 (40, 40)
- 若无焦点窗口，使用屏幕中心
- 添加屏幕边界检测，防止新窗口出现在屏幕外

**修改位置**：`src-tauri/src/window_manager.rs` — `create_note()` 函数

#### 修复点 2：确保新窗口获得焦点并置顶

**当前逻辑**：`build_note_window()` 已设置 `.focused(true)`，但可能不够。

**优化方案**：
- 创建窗口后显式调用 `win.set_focus()` 和 `win.set_always_on_top(true)` + 延迟取消置顶（若非置顶模式）
- 或使用 `win.show()` + `win.set_focus()` 组合确保窗口可见

**修改位置**：`src-tauri/src/window_manager.rs` — `build_note_window()` 和 `create_note()` 函数

#### 修复点 3：前端 IPC 参数命名对齐

**当前问题**：前端调用 `invoke('cmd_create_note', { color: note.dataset.color })`，而 Tauri v2 的 `#[tauri::command]` 宏默认使用 snake_case 参数名。

**确认**：当前 `cmd_create_note` 的参数名为 `color`（单字），snake_case 与 camelCase 一致，不存在映射问题。但需验证多字参数（如 `itemsJson` → `items_json`）是否正确映射。

**修改位置**：如需修改，涉及 `src/index.html` 前端调用参数名

## 8. 非功能需求

- **拖拽流畅度**：窗口移动帧率 ≥ 30fps，无明显延迟感
- **新建响应**：点击到新窗口出现 ≤ 500ms
- **兼容性**：修复方案需兼容 Windows 10/11，WebView2 最新版及最近两个大版本
- **回归**：修复后置顶、最小化、颜色切换、内容编辑等现有功能不受影响
